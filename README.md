## 1. 项目概述

### 1.1 问题

同花顺远航版客户端积累了长期人工维护的测试用例，存在以下问题：

- **组件过期**: 用例引用的 UI 组件在当前版本中已不存在
- **预期不一致**: 预期结果与实际界面表现不符
- **描述不清**: 操作步骤不够清晰，无法复现
- **无法执行**: 由于环境、数据或组件缺失导致无法执行

### 1.2 方案

本方案的核心思路是 **"让 AI 去跑一遍测试用例"**：

1. **AI 解析用例** → 将自然语言描述转为结构化的 Playwright 操作指令
2. **Playwright 自动执行** → 在真实客户端上执行操作（点击、输入、截图）
3. **AI 分析结果** → 根据截图 + 执行日志，判断通过/失败，生成清洗标签
4. **输出清洗报告** → Excel 结果 + Markdown/HTML 报告 + 修复建议

### 1.3 "清洗"的定义

在本工程中，"清洗"包括以下维度：

| 维度 | 检查内容 | 产出 |
|------|---------|------|
| 可执行性 | 用例步骤能否完整执行 | 通过标签 `可执行` / `不可执行` |
| 组件存在性 | 涉及的 UI 组件是否存在 | 标签 `组件缺失` |
| 预期一致性 | 实际结果是否匹配预期 | 标签 `预期不一致` |
| 步骤完整性 | 步骤是否足够清晰详细 | 标签 `步骤不足` / `描述不清` |
| 正确性验证 | 预期结果是否合理 | 通过/未通过 判断 + 修复建议 |

## 3. 工程架构

### 3.1 模块职责

| 模块 | 文件 | 职责 |
|------|------|------|
| **数据模型** | `src/__init__.py` | TestCase, PlaywrightAction, CleaningReport 等数据类 |
| **Excel 解析** | `src/parser/excel_parser.py` | 读取 Excel, 解析列映射, 构建 TestCase 列表 |
| **AI 客户端** | `src/ai/llm_client.py` | 封装 OpenAI/DeepSeek API, 支持 mock 模式 |
| **AI 提示词** | `src/ai/prompts.py` | 步骤解析、操作生成、结果分析、报告摘要的提示词模板 |
| **操作生成器** | `src/engine/action_generator.py` | AI 模式/规则模式 两种方式生成 Playwright 操作 |
| **Playwright 执行** | `src/engine/playwright_runner.py` | 连接浏览器、执行操作、截图、收集日志 |
| **清洗核心** | `src/cleaner/case_cleaner.py` | 串联完整流程, demo 模式加载预置结果 |
| **报告生成** | `src/reporter/report_generator.py` | 生成 Excel/Markdown/HTML 三种格式报告 |

### 3.2 两种运行模式

#### 演示模式 (`demo`)
- **无需 API Key**
- **无需真实客户端**
- 使用内置的预置清洗结果（来自真实执行后 AI 分析的记录）
- 适用于演示、CI、快速了解工具能力

#### 完整模式 (`full`)
- **需要 OpenAI API Key**
- **需要运行中的同花顺远航版客户端**
- **需要 Playwright 浏览器**
- 真实执行每个用例步骤
- AI 实时分析执行结果

## 4. 安装与运行

### 4.1 环境要求

- Python 3.10+
- pip
- (完整模式) 同花顺远航版客户端
- (完整模式) OpenAI API Key

### 4.2 安装步骤

```bash
# 1. 克隆/解压项目
# cd PlaywrightTest

# 2. 安装 Python 依赖
pip install -r requirements.txt

# 3. 安装 Playwright 浏览器（仅完整模式需要）
playwright install chromium

# 4. (可选) 设置 API Key（仅完整模式需要）
# Windows PowerShell:
$env:OPENAI_API_KEY = "sk-xxx"
# Linux/macOS:
export OPENAI_API_KEY=sk-xxx
```

### 4.3 运行

```bash
# === 演示模式（推荐首次使用） ===
python main.py demo
# 输出:
# - outputs/cleaned_cases.xlsx    清洗后的 Excel
# - outputs/cleaning_report.md    Markdown 报告
# - outputs/cleaning_report.html  HTML 报告

# === 完整模式（需要 API Key + 客户端） ===
python main.py full

# === 仅解析 Excel ===
python main.py parse

# === 指定 Excel 文件 ===
python main.py demo --excel-path "path/to/你的文件.xls"

# === 指定输出目录 ===
python main.py demo --output-dir "my_results"
```

### 4.4 配置说明

编辑 `config.yaml` 自定义行为：

```yaml
# 切换 AI 服务商（DeepSeek 示例）
ai:
  provider: "deepseek"
  model: "deepseek-chat"
  base_url: "https://api.deepseek.com/v1"
  api_key_env: "DEEPSEEK_API_KEY"
  temperature: 0.1

# Playwright 配置
playwright:
  headless: true        # 无头模式（无显示器也可运行）
  timeout: 30000        # 单步超时
  viewport_width: 1280
  viewport_height: 800
```

## 5. 运行结果

### 5.1 输出文件说明

运行 `python main.py demo` 后，`outputs/` 目录下会生成：

| 文件 | 格式 | 内容 |
|------|------|------|
| `cleaned_cases.xlsx` | Excel | 原始用例 + 清洗结果（是否通过、未通过原因、标签、AI建议） |
| `cleaning_report.md` | Markdown | 完整分析报告：统计、标签分布、AI 摘要、逐用例详情 |
| `cleaning_report.html` | HTML | 可视化网页报告 |

### 5.2 清洗结果概览

以下是演示模式在 10 条样例用例上的清洗结果摘要：

| 指标 | 数值 |
|------|------|
| 总用例数 | 10 |
| ✅ 通过 | 5 |
| ❌ 未通过 | 5 |
| 📊 通过率 | 50.0% |
| 🚫 不可执行 | 5 |

### 5.3 标签分布

| 标签 | 数量 | 说明 |
|------|------|------|
| 通过 | 5 | 用例正常执行且符合预期 |
| 可执行 | 5 | 用例在当前版本中可执行 |
| 未通过 | 5 | 用例执行失败或验证不通过 |
| 不可执行 | 5 | 用例因组件缺失等原因无法执行 |
| 组件缺失 | 5 | 涉及的 UI 组件已下线或不存在 |
| 预期不一致 | 2 | 预期结果与当前版本不符 |

### 5.4 关键发现

1. **50% 用例通过**: 基础布局、默认尺寸、标签切换、列表显示、窗口缩放功能正常
2. **50% 用例不可执行**: 5 条用例均因引用的"示例板块"组件在当前版本不存在而无法执行
3. **组件生命周期管理**: "示例板块"组件被移除后，依赖该组件的所有用例均需更新或废弃
4. **预期结果耦合**: 部分用例预期结果与具体数据（标签名、菜单名）强耦合，版本迭代时容易失效

### 5.5 逐用例详情

#### ✅ 通过的用例

| 用例名称 | 原因 |
|---------|------|
| TC_组件窗口_基础布局 | 窗口布局要素完整：标题栏、内容区、菜单按钮均显示正常 |
| TC_组件窗口_默认尺寸 | 窗口以默认尺寸打开，内容区域完整显示，无遮挡 |
| TC_导航栏_标签显示 | 导航栏标签正常显示，选中状态、高亮、切换刷新均符合预期 |
| TC_列表区域_基础显示 | 左侧分类列表、右侧明细区域显示正常，分类切换后明细刷新正确 |
| TC_窗口缩放_横向拉伸 | 窗口拉伸后内容区域自适应扩展，控件布局正常，无重叠 |

#### ❌ 未通过的用例

| 用例名称 | 原因 | 清洗标签 |
|---------|------|---------|
| TC_组件入口_打开板块 | 不存在"示例板块"组件入口 | 未通过、不可执行、组件缺失 |
| TC_导航栏_默认选中项 | 无法验证默认选中标签（组件缺失）；预期硬编码了具体标签名 | 未通过、不可执行、组件缺失 |
| TC_右上角菜单_分组设置 | 组件缺失导致无法验证；菜单项名称可能已变更 | 未通过、不可执行、组件缺失、预期不一致 |
| TC_窗口最小宽度_校验 | 目标组件不存在，无法执行 | 未通过、不可执行、组件缺失 |
| TC_列表刷新_切换分类 | 目标组件不存在；预期结果跨模块耦合 | 未通过、不可执行、组件缺失、预期不一致 |

#### 修复建议汇总

1. **组件替换**: 将引用"示例板块"的用例统一替换为当前版本实际存在的业务组件（如"板块指数"、"自选股"等）
2. **预期结果参数化**: 将预期结果中的硬编码标签名、菜单名改为相对描述（如"存在默认选中项"代替"选中【地域指数】"）
3. **解耦跨模块依赖**: TC_列表刷新_切换分类 中与"行情中心"的一致性校验应解耦
4. **量化预期**: TC_窗口最小宽度_校验 应给出具体的最小像素宽度值
5. **废弃处理**: 确认"示例板块"组件是否已永久下线，若是则标记相关用例为废弃
6. **定期清洗**: 建议每个版本/每月执行一次自动清洗，防止用例腐化

## 6. AI 使用方式详解

### 6.1 AI 在工程中的角色

本工程使用 AI 来完成传统自动化难以完成的"理解与判断"工作：

```
传统自动化                AI 增强自动化
───────────              ──────────────
人工编写定位器           AI 理解自然语言描述生成操作
硬编码断言               AI 看图判断是否符合预期
规则匹配失败原因           AI 分析失败原因并给出修复建议
人工分析报告              AI 自动生成结构化报告
```

### 6.2 AI 调用的 4 个环节

#### 环节 1: 步骤解析 (Step Parsing)

```
输入: "1、打开【通用工具】-【示例板块】组件\n2、观察窗口整体区域"
  │
  ▼ LLM
输出: [
  {"index": 1, "action": "click", "target": "通用工具", ...},
  {"index": 2, "action": "click", "target": "示例板块", ...},
  {"index": 3, "action": "screenshot", "target": "窗口整体", ...}
]
```

**为什么用 AI**: 自然语言描述有多种表达方式（"打开"、"点击进入"、"进入"），传统规则匹配覆盖不全。

#### 环节 2: 操作指令生成 (Action Generation)

```
输入: 结构化的步骤 + 用例上下文
  │
  ▼ LLM
输出: [
  {"action_type": "click", "target": "通用工具", "timeout": 10000},
  {"action_type": "wait", "value": 2000},
  {"action_type": "click", "target": "示例板块", "timeout": 10000},
  {"action_type": "screenshot"},
  {"action_type": "assert", "target": "标题栏、内容区、菜单按钮"}
]
```

**为什么用 AI**: 不同操作需要不同策略（等待时间、定位方式、错误处理），AI 可以根据上下文做出合理判断。

#### 环节 3: 结果分析 (Result Analysis)

```
输入: 执行日志 + 截图 + 预期结果
  │
  ▼ LLM
输出: {
  "passed": false,
  "reason": "当前版本客户端中不存在「示例板块」组件入口",
  "tags": ["未通过", "不可执行", "组件缺失"],
  "suggestion": "确认组件是否已下线; 替换为实际存在的业务板块"
}
```

**为什么用 AI**: 传统自动化只能判断"元素找不到"→报错，但无法理解"为什么找不到"，也无法给出修复建议。

#### 环节 4: 报告摘要 (Report Summary)

```
输入: 统计数据 + 标签分布 + 失败用例详情
  │
  ▼ LLM
输出: 专业的自然语言报告摘要 + 改进建议
```

**为什么用 AI**: 将冷冰冰的统计数据转化为可读的分析报告。

### 6.3 降级策略

本工程设计了 **无 AI 降级路径**：

- 如果检测不到 API Key → 自动切换到"规则引擎模式"
- 规则引擎通过正则匹配操作描述中的关键词（"打开"、"点击"、"观察"等）来生成基本操作
- 演示模式使用 "预置 AI 分析结果"，展示完整效果，同时零外部依赖

### 6.4 提示词工程

详见 `src/ai/prompts.py`。提示词设计原则：

1. **角色明确**: 每个提示词定义了明确的 AI 角色（测试工程师、Playwright 专家、结果分析员）
2. **结构化输出**: 要求 JSON 格式输出，便于程序解析
3. **上下文传递**: 将用例名称、前置条件、操作描述等上下文完整传递给 AI
4. **约束引导**: 通过"可用操作类型及参数说明"引导 AI 生成可执行的操作
5. **案例如下**: 在提示词中嵌入示例，帮助 AI 理解预期输出格式

## 7. 当前局限性

### 7.1 技术局限

| 局限 | 说明 | 改进方向 |
|------|------|---------|
| **桌面应用自动化** | 同花顺远航版是 Windows 桌面应用（可能基于 Electron/WebView），Playwright 原生面向 Web 应用。自动化需要额外的窗口层级处理。 | 集成 `pywinauto` 或 `uiautomation` 进行窗口管理 |
| **单线程顺序执行** | 当前 demo 为顺序执行，大规模用例清洗耗时长 | 引入 asyncio 并发 + 多窗口/多实例并行 |
| **AI 不确定性** | LLM 输出具有概率性，低 temperature 可减少但无法消除 | 引入多次采样 + 投票机制；关键判断人工复核 |
| **视觉分析依赖截图** | AI 通过静态截图判断结果，无法感知动画、过渡效果、时序问题 | 录屏 + 逐帧分析；增加交互式验证 |
| **元素定位精度** | 文本匹配可能定位到错误元素（同名不同层） | AI 辅助生成更精准的选择器 |

### 7.2 功能局限

| 局限 | 说明 | 改进方向 |
|------|------|---------|
| **只覆盖 UI 层面** | 仅验证前端表现，不校验后端数据 | 集成 API 检验（HTTP 拦截 + 数据库查询） |
| **预处理依赖** | 用例引用的数据、环境依赖需人为准备 | 增加环境自检 + 数据 Mock 能力 |
| **组件库知识有限** | AI 不了解同花顺特定组件库和业务术语 | 提供组件库文档作为 context；微调私有大模型 |
| **无版本对比** | 不能检测"以前通过、现在失败"的回归 | 增加历史结果数据库 + 版本间差异对比 |

### 7.3 Demo 局限

- **演示数据规模小**: 仅 10 条用例，实际场景可能有上万条
- **预置结果覆盖有限**: demo 模式的结果仅覆盖当前 10 条用例的已知情况
- **未覆盖所有用例类型**: 数据校验、接口测试、性能测试等类型的用例未包含
